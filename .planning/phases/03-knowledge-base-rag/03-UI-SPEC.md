# Phase 3 — UI Design Contract

> Visual and interaction contract for Knowledge Base + RAG. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Reuse-first phase.** The design system (tokens, spacing, typography, color) is fully established by Phase 1/2 and locked in `src/index.css`. This contract specifies only what is NEW or ambiguous in Phase 3: citation rendering (inline `[n]` markers + source cards), the two-pane KB browser layout + empty state, indexed-status badge states, ingestion progress surfaces, the KB-scope selector, and the notes editor chrome. The chat shell, sidebar, resizable split, and existing tokens are REUSED unchanged.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding (Tailwind: `gap-1`, `px-1`) |
| sm | 8px | Compact element spacing (Tailwind: `gap-2`, `p-2`) |
| md | 16px | Default element spacing (Tailwind: `gap-4`, `p-4`) |
| lg | 24px | Section padding (Tailwind: `gap-6`, `p-6`) |
| xl | 32px | Layout gaps (Tailwind: `gap-8`, `p-8`) |
| 2xl | 48px | Major section breaks (Tailwind: `gap-12`, `p-12`) |
| 3xl | 64px | Page-level spacing (Tailwind: `gap-16`, `p-16`) |

**Exceptions (inherited from Phase 1/2):**
- Sidebar width: 48px (`w-12`).
- Resizable two-pane split: reuse Phase 2 `react-resizable-panels`; left tree pane default 280px, resizable 20%–40% of viewport (matches the conversation-list constraint).
- Control height: 36px (`h-9`) for inputs/buttons/selects; 28px (`h-7 w-7`) for hover/row icon actions — matches `MessageInput`/`MessageBubble`.

**Phase 3-specific spacing:**
- KB browser pane content padding: 16px (`p-4`).
- Folder tree row height: 28px (`h-7`), 4px row gap (`gap-1`) — dense, VSCode-sidebar feel via `react-arborist`.
- Items table row padding: 8px vertical (`py-2`), 16px horizontal (`px-4`) per cell; faceted-filter toolbar gap 8px (`gap-2`).
- Indexed-status `Badge`: 0px vertical / 6px horizontal padding, 16px height (`h-4 px-1.5 py-0`) — matches the Phase 2 model badge exactly.
- Source cards (below answer): 12px internal padding (`p-3`), 8px gap between cards (`gap-2`), 8px top margin from answer body (`mt-2`).
- Inline citation marker `[n]`: rendered as `<sup>` span, 2px horizontal padding (`px-0.5`), no added line-height shift.
- Drop zone (empty state): 48px internal padding (`p-12`), centered; 16px gap between headline, glyph, drop zone, and fallback actions (`gap-4`).
- Global progress panel: 12px padding (`p-3`), 8px gap between bar and "3 de 8" label (`gap-2`).
- Notes editor: 16px outer padding (`p-4`) around the CodeMirror surface; editor gutter/line padding governed by CodeMirror defaults (do not override densely — keep readable).

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 (regular) | 1.5 |
| Label | 12px | 500 (medium) | 1.4 |
| Heading (pane title / empty-state headline) | 14px–16px | 500 (medium) | 1.4 |
| Code / Markdown editor (monospace) | 13px | 400 (regular) | 1.6 |

**Notes (consistency with Phase 2 — do NOT introduce new sizes):**
- Body weight 400 for content (table cells, note text, source-card filenames, answer body).
- Weight 500 for emphasis: pane titles, column headers, faceted-filter labels, empty-state headline, badge text.
- Tree rows, table cells, and badges use `text-sm`/`text-xs` exactly as Phase 2 — no third weight, no display size.
- Notes editor (CodeMirror) renders raw Markdown in a 13px monospace font at 1.6 line-height (matches the chat code-block convention); inline Markdown syntax highlighting via `@codemirror/lang-markdown` gives the Obsidian live-preview feel WITHOUT mutating the text (D-08).
- Citation `[n]` markers inherit body size rendered as superscript (`text-xs`, `font-medium`), styled as a link affordance (see Color).

**Font stack:** Inherited from Phase 1 (system font stack via Tailwind v4 defaults). Monospace for editor/code via the default mono stack.

## Color

| Role | Value (dark mode) | Usage |
|------|-------------------|-------|
| Dominant (60%) | `hsl(224 71.4% 4.1%)` | App/pane background, table surface, editor background |
| Secondary (30%) | `hsl(215 27.9% 16.9%)` | Tree pane background, source-card surface, hovered table rows, progress-panel surface |
| Accent (10%) | `hsl(263.4 70% 50.4%)` (violet, default) | **Reserved — see explicit list below** |
| Destructive | `hsl(0 62.8% 30.6%)` (dark) / `hsl(0 72.2% 50.6%)` (light) | `failed` badge, delete-item action, terminal-failure toast accent |

**Accent reserved for (explicit — never "all interactive elements"):**
1. Inline citation `[n]` marker text + hover/focus state (the one accent affordance in the answer body).
2. KB-scope selector "active/on" state in `MessageInput` (e.g., toggled "Usar KB" pill background `bg-accent text-accent-foreground`, mirroring the send button).
3. "Adicionar" / "Escolher arquivos" primary buttons in the empty state (primary CTAs).
4. Active/selected folder-tree row indicator and focus rings (`ring-2 ring-ring` — `--ring` shares the accent hue).
5. Global progress bar fill (the `Progress` indicator track — `bg-primary`).

**NOT used for:** indexed-status badges (pending/indexing/indexed — these use neutral/`muted` tones; only `failed` uses destructive), type icons, table headers, tree chevrons, source-card borders, generic table/row text, faceted-filter chips.

**Indexed-status badge color semantics (D-11/D-12) — neutral-by-default, accent-free:**
| Status | Badge variant / classes | Meaning |
|--------|-------------------------|---------|
| `pending` | `variant="secondary"`, `bg-muted text-muted-foreground` | Queued, not yet processed |
| `indexing` | `variant="secondary"`, `bg-muted text-muted-foreground` + subtle pulse / spinner glyph (`Loader2` animate-spin, 14px) | In progress |
| `indexed` | `variant="outline"` or `secondary` with `text-muted-foreground` + check glyph (`Check`, 14px) | Done — quiet, low-emphasis (success is the expected state, not celebrated) |
| `failed` | `variant="destructive"` (`bg-destructive text-destructive-foreground`) + `AlertCircle` glyph | Terminal error; carries Retry action + tooltip with `error_reason` |

> Rationale: badges are status meta, not CTAs. Following the Phase 2 rule that the model badge uses `bg-muted` (NOT accent), KB status badges stay muted so accent stays scarce; only `failed` earns the loud destructive color because it demands action.

**Other surface semantics:**
- Source cards: `bg-secondary` surface, `border-border`, `text-foreground` filename (weight 500) + `text-muted-foreground` section/page meta (weight 400).
- Drop zone: dashed `border-2 border-dashed border-border`, `text-muted-foreground` body; on drag-over → `border-accent` + `bg-accent/5` (mirrors the Phase 2 `MessageInput` drag overlay `ring-2 ring-accent`).
- Error/failure inline + toast: `text-destructive` on `bg-destructive/10 border-destructive/20` (reuse the Phase 2 inline-error treatment).

**Light mode:** Inherited from Phase 1 tokens; phase optimizes for dark mode but must not break light mode (tokens are theme-aware).

## Interaction Patterns

### Ingestion (file / drop / picker — KB-01)
1. User drops files on the drop zone OR clicks "Escolher arquivos" (Rust file dialog).
2. Each item appears in the table immediately with a `pending` badge.
3. Rust streams `IndexProgress` via Channel → `indexingStore` → badge transitions `pending → indexing → indexed` (or `failed`).
4. Batch imports show the global `Progress` panel ("Indexando 3 de 8"); indeterminate bar when chunk total is unknown.
5. On completion, badges settle to `indexed`; on failure, `failed` badge + "Reindexar" + tooltip reason.

### URL ingestion (KB-04)
1. User pastes a URL in the empty-state/table-toolbar input and clicks "Adicionar".
2. Sonner toast: "URL adicionada. Indexando..."; row appears with `pending`/`indexing` badge.
3. On the `Indexed` Channel event → toast "URL indexada e pronta para consulta." (event-driven, not a timer — KB-04 "within one minute").
4. On failure (dynamic page / timeout) → `failed` badge + destructive toast with the PT-BR reason.

### Retry / re-index (D-12)
1. `failed` row shows a "Reindexar" action (button or row menu).
2. Click → Rust idempotently deletes existing chunks/vectors, re-embeds; badge returns to `indexing`.

### RAG grounding in chat (D-01/D-03)
1. In `MessageInput`, user toggles "Usar KB" beside the model picker (per-message).
2. When on, the message is grounded: retriever runs (BM25 + sqlite-vec + RRF), top chunks injected, citation prompt added.
3. AI answer streams via the existing pipeline; inline `[n]` markers + source cards render in the bubble.
4. Toggle is per-message — a conversation can mix grounded and ungrounded messages.

### Notes editing (KB-03 / D-07/D-08)
1. User selects a note in the tree (or creates "Nova nota").
2. CodeMirror opens with raw Markdown; inline syntax highlighting gives live-preview feel.
3. Edits save the exact text typed — no normalization/serialization round-trip (D-08).
4. Saved notes re-embed and become retrievable alongside files.

### Faceted filtering (KB-05 / D-10)
1. Toolbar exposes `Tipo` and `Status` faceted filters.
2. Selecting facet values narrows the table in real time.
3. No results → "Nenhum item encontrado para esse filtro."

### Delete item
1. Right-click row/tree node or row menu → "Excluir".
2. Alert-Dialog confirms: "Excluir item da base? Esta ação não pode ser desfeita."
3. Confirm → soft/hard delete + chunk/vector cleanup in Rust.

## States

### Indexing item states (single source: `indexingStore`, reconciled with DB `status` column on reload — D-11)
- **pending:** muted badge "Pendente", queued.
- **indexing:** muted badge "Indexando" + spinner glyph; contributes to global progress count.
- **indexed:** quiet muted badge "Indexado" + check glyph.
- **failed:** destructive badge "Falhou" + `AlertCircle` + "Reindexar" action + tooltip `error_reason`.
- **stuck-on-reload reconcile:** any item left `indexing` at app start (dropped Channel event) → mark `failed`, offer Reindexar (Pitfall 5 in RESEARCH.md).

### Browser states
- **Empty (no items):** drop-zone empty state (D-13).
- **Populated:** tree + table.
- **Filtered, no match:** "Nenhum item encontrado para esse filtro."
- **First run (model not cached):** one-time "Baixando modelo de embeddings (uma única vez)..." progress (reuses the `Progress` infra).

### Notes editor states
- **Empty note:** placeholder/empty CodeMirror, title "Nova nota".
- **Editing:** raw Markdown, unsaved indicator optional; autosave or explicit save (implementation discretion — must not mutate text).

### Citation states
- **Grounded answer with reliable markers:** inline `[n]` + source cards.
- **Grounded answer, weak markers (D-06 fallback):** source cards only, no inline markers.
- **Ungrounded answer:** no citations, no cards (normal chat).

## Accessibility

- **Keyboard navigation:** tree (`react-arborist` arrow-key nav), table rows, faceted filters, and the KB-scope toggle are all focusable with visible focus rings (`ring-2 ring-ring`). Editor is keyboard-first (CodeMirror).
- **Icon-only controls:** type icons and status badges carry `aria-label` / `title` (e.g., status announced as "Indexado"); the `failed` badge's tooltip reason is also exposed to screen readers.
- **Color is not the only signal:** each status badge pairs color with text label + a glyph (check / spinner / alert) so colorblind users can distinguish states.
- **Color contrast:** all text passes WCAG AA (4.5:1 body, 3:1 large/badge text). `failed` destructive badge verified against `destructive-foreground`.
- **Citations:** inline `[n]` markers are real buttons/links (`role`/`aria-label` "Fonte 1") — keyboard-activatable, not click-only spans.
- **Toasts (sonner):** announced via aria-live; non-blocking; terminal failures also persist in the row badge so the information isn't toast-only.

- **Resizable panel:** no animation (immediate, persisted).
